## M1：登录 Session Token 改造

### 1.1 问题陈述

**当前**：`POST /auth/login` 验证密码后直接返回 `{"token": request.password}`，前端把这个明文密码当作 token 存 `localStorage`，后续所有面板 API 都用 `Authorization: Bearer <密码>`。

**风险**：
1. token 泄露（浏览器 devtools、日志、Referer、第三方 JS）= 密码泄露
2. token 永不过期，长期有效
3. 无法主动吊销（除非改密码）
4. 控制台 token = 面板密码 = API 密码（共用 `PASSWORD` 时）

**目标**：登录成功后返回独立的服务端 session token，原密码不再直接作为 token 使用。

### 1.2 数据模型

### 1.9 迁移与兼容性

| 场景 | M1 上线后行为 |
|---|---|
| 老客户端用 `Authorization: Bearer <密码>` | 过渡期 fallback 通过，30 天后失效 |
| 老客户端 localStorage 里的明文密码 | 过渡期仍可用，30 天后需重新登录获取 session |
| 新客户端登录 | 拿到 session_id，后续所有请求用 session |
| 用户改密码 | 所有旧 session 失效，强制重登 |
| 服务重启 | session 在 MongoDB，不丢失；环境变量密码改了则旧 session 失效 |

### 1.10 测试要点

1. **登录**：返回的 `token` 是 43 字符的 `token_urlsafe` 字符串，不是密码
2. **鉴权**：用新 session 调 `/creds/list` `/config/get` `/logs/clear` 均通过
3. **过期**：手动把 `expires_at` 改成过去时间，鉴权应返回 401
4. **续期**：连续调用 API，`last_active_at` 应更新
5. **登出**：`POST /auth/logout` 后再用该 session 鉴权应 401
6. **fallback**：用明文密码作 Bearer token，过渡期内应通过
7. **改密码联动**：`/config/save` 改密码后，旧 session 全部失效
8. **多 worker**：session 在 MongoDB，多 worker 下共享一致

### 1.11 风险点

1. **session_id 泄露**：相比密码泄露，影响有限（有 TTL、可远程踢除、不影响 API 密码功能）
2. **DB 抖动**：session 查询失败时 fallback 到密码，避免全员登出；但日志要告警
3. **fallback 期暴露**：30 天 fallback 期内，攻击者拿到明文密码仍可直连；建议在面板日志里统计 fallback 命中次数，若 30 天后仍有流量则延长过渡期并告警
4. **环境变量密码场景**：改环境变量密码需重启服务，所有 session 因 `password_hash` 不匹配而失效（需在 `verify_panel_token` 里加这个判断 —— 见下方补充）

## M4：密码加密落盘（argon2）

### 2.1 问题陈述

**当前**：密码以明文存于环境变量（运行时内存）和 DB 配置表（通过 `/config/save` 写入）。DB dump 泄露 → 密码明文泄露。

**目标**：DB 中只存 argon2 hash，无法反推出明文密码。

### 2.2 前置依赖

**M1 必须先完成**。理由：
- argon2 验证单次约 100ms，若每次 API 调用都跑 argon2，高频 API 会被拖垮
- M1 完成后，API 调用走 session 校验（O(1) DB 查询），密码验证只在登录时跑一次 argon2，开销可接受

### 2.3 算法与依赖

- **算法**：argon2id（OWASP 推荐）
- **库**：`argon2-cffi`（`pip install argon2-cffi`）
- **参数**：使用 `argon2.PasswordHasher()` 默认参数（time_cost=3, memory_cost=64MB, parallelism=4），单次约 100ms

**依赖添加**：`pyproject.toml` 的 `dependencies` 加 `argon2-cffi>=23.1.0`

### 2.4 存储变更

#### 2.4.1 DB 配置表

密码字段从明文改为 argon2 hash 字符串（格式 `$argon2id$...`）。

**识别方式**：hash 以 `$argon2id$` 或 `$argon2i$` / `$argon2d$` 开头；明文则不是。

#### 2.4.2 环境变量

新增 hash 形式环境变量，优先级（高 → 低）：
PANEL_PASSWORD_HASH > PANEL_PASSWORD > 默认弱密码拒绝 API_PASSWORD_HASH > API_PASSWORD > 默认弱密码拒绝 SERVER_PASSWORD_HASH > SERVER_PASSWORD > 默认弱密码拒绝


Plain Text


**理由**：环境变量在 HF Space Secret 中存储，明文密码出现在环境变量本身是可接受的（Secret 加密存储），但提供 hash 形式让 paranoid 用户可以完全不落明明文密码。

## L1：凭证文件加密（建议暂缓）

### 3.1 问题陈述

**当前**：OAuth 凭证（`access_token` / `refresh_token` / `client_secret`）明文存于 MongoDB。

### 3.2 威胁模型分析

| 攻击场景 | 加密是否有效 |
|---|---|
| 攻击者拿到 HF Space 完全控制（含 Secret） | **无效**（密钥也在 Secret 中） |
| 攻击者仅 dump MongoDB（无 HF Secret） | **有效**（无密钥无法解密 refresh_token） |
| 攻击者通过面板 API 拉取凭证 | **无效**（API 返回明文，加密只在落盘层） |
| 内部 DBA 接触数据库 | **有效**（DBA 看不到明文 refresh_token） |

### 3.3 实施方案（如需启用）

#### 3.3.1 算法与依赖

- **算法**：Fernet（`cryptography` 库，AES-128-CBC + HMAC-SHA256）
- **依赖**：`cryptography>=42.0.0`（`pyproject.toml`）
- **主密钥**：HF Space Secret 新增 `CREDENTIAL_ENCRYPTION_KEY`（32 字节 base64，`Fernet.generate_key()` 生成）

#### 3.3.2 改造点

**文件**：`src/storage_adapter.py` 的 credential save/load 路径

```python
from cryptography.fernet import Fernet import os

_fernet = None

def _get_fernet() -> Fernet: global _fernet if _fernet is None: key = os.getenv("CREDENTIAL_ENCRYPTION_KEY") if not key: raise RuntimeError("CREDENTIAL_ENCRYPTION_KEY 未配置，无法加解密凭证") _fernet = Fernet(key) return _fernet

#### 3.3.4 密钥丢失处理

**密钥丢失 = 所有 OAuth 凭证失效**，需重新走 OAuth 流程重新登录。建议：
- 密钥备份到另一个安全位置（如密码管理器）
- 文档明确警告"密钥丢失需重新 OAuth"

### 3.4 评估结论：建议暂缓

**理由**：
1. 用户当前 MongoDB 连接字符串本身已是 HF Secret，泄露连接字符串的场景下大概率也泄露了加密密钥（同一信任域）
2. 加密主要防护"DBA 查库"和"DB dump 单独泄露"两个边缘场景
3. 真正高价值的攻击路径是"拿到整个 HF Space"，加密对此无效
4. 实施工作量中等但需引入密钥管理流程，收益/成本比偏低

**优先实施条件**：
- MongoDB 改为共享/托管服务（如 MongoDB Atlas），且 HF Space 与 DB 不在同一信任域
- 或团队多人运维，DBA 不应见到 OAuth refresh_token
- 或合规要求（如等保）强制要求静态加密

### 3.5 测试要点（如实施）

1. **新写入**：保存凭证 → DB 中 access_token 是密文 → load 后是明文
2. **迁移**：DB 中已有明文凭证 → 启动 → 自动加密 → 日志告警 → load 正常
3. **密钥错误**：换一个 `CREDENTIAL_ENCRYPTION_KEY` → load 抛异常 → 凭证失效
4. **未配置密钥**：`CREDENTIAL_ENCRYPTION_KEY` 为空 → 启动报错或 fallback 明文（取决于策略）
5. **兼容**：旧凭证（无 `_encrypted` 字段）仍可读取
